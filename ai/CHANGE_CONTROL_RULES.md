\# CHANGE\_CONTROL\_RULES.md



\## 1. CONTINUATION-ONLY MODE



This project is an actively developed codebase.



AI assistance is restricted to:

\- Extending existing\_attach architecture

\- Completing missing but clearly implied functionality

\- Fixing bugs

\- Wiring existing components together



The AI MUST NOT:

\- Redesign the application

\- Introduce new frameworks

\- Replace the tech stack

\- Reorganize folders

\- Rename files

\- Delete files

\- “Clean up” or refactor working code



Unless explicitly instructed.



---



\## 2. LIMITED FILE MODIFICATION



Without explicit permission, the AI may:

\- Modify at most TWO (2) files per change set



If more files are required, the AI MUST:

\- Stop

\- Explain why

\- Ask for approval before proceeding



---



\## 3. NO SILENT REFACTORING



The AI MUST NOT:

\- Perform “improvements” not requested

\- Normalize naming conventions

\- Optimize code for performance or elegance

\- Apply best-practice refactors unless asked



Correctness and safety take priority over cleanliness.



---



\## 4. EXPLAIN BEFORE WIDE CHANGES



If a requested change could affect:

\- Application flow

\- User-facing language

\- Legal safety posture

\- Compliance constraints



The AI MUST:

\- Explain the impact in plain English

\- Wait for explicit approval

\- Proceed only after approval



---



\## 5. SINGLE-AI AUTHORITY



At any given time:

\- Only ONE AI tool is authorized to edit files

\- Other AI tools are advisory only



The currently authorized editor must be explicitly acknowledged by the user.



---



\## 6. FAILURE MODE



If there is ambiguity about:

\- Rules

\- Scope

\- Authority

\- Safety



The AI MUST:

\- Stop

\- Ask a clarifying question

\- Make NO changes until clarified
